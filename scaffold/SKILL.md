# Scaffold Branch — Scaffold Workflow Guide

## Purpose

Transform documented SDLC requirements (from the Knowledge branch) into
a working project skeleton: directory structure, configuration files, CI/CD
pipelines, database schemas, API routes, event handlers, and test stubs.

## Environment

- **Base directory:** `.` (repository root)
- **Scaffold CLI:** `python3 scaffold/build_chain/cli.py <command> [args]`
- **Knowledge CLI (read-only):** `python3 knowledge/sdlc_chain/cli.py <command> [args]`
- **Knowledge artifacts:** `knowledge/artifact/[subdir]/*.json`
- **Body output directories:**
  - `scaffold/src/` — generated source code
  - `scaffold/config/` — deployment and infrastructure configs
  - `scaffold/ci/` — CI/CD pipeline files
  - `scaffold/db/` — database schemas and migrations
  - `scaffold/tests/unit/` — unit test stubs
  - `scaffold/tests/performance/` — performance test scaffolds
- **Temp files:** `scaffold/.tmp/` — session-scoped, cleaned on exit

## Core Workflow

### Step 1 — Discover Knowledge Artifacts

Run the knowledge CLI to see what documentation exists:

```bash
python3 knowledge/sdlc_chain/cli.py list-existing
```

This returns JSON with `doc_type`, `document_id`, `relative_path`, and
`directory` for each artifact. Group by layer:

| Layer | Doc Types | Scaffold Action |
|-------|-----------|-----------------|
| 3 Architecture | HLD, TSI, CICD, DBAD | Project layout, CI/CD, DB schemas |
| 3 Contracts | API, AEC, DC, DBC, MDC | API stubs, event handlers, data models |
| 4 Testing | UT, NFTS, DG | Test stubs, deployment configs |

### Step 2 — Check Scaffold Status

Run the scaffold CLI to see what has been scaffolded vs what is pending:

```bash
python3 scaffold/build_chain/cli.py scaffold-status
```

This cross-references knowledge artifacts against existing scaffold output
files, checking traceability headers for source doc IDs and versions.

### Step 3 — Scaffold in Layer Order

Always process in dependency order:

1. **Architecture first** — `/scaffold-project` (HLD + TSI)
2. **CI/CD and deploy** — `/scaffold-cicd` (CICD + DG)
3. **Database** — `/scaffold-db` (DBAD + DC)
4. **API and events** — `/scaffold-api` (API + AEC)
5. **Tests** — `/scaffold-tests` (UT + NFTS)

For each scaffold:
1. Read the relevant knowledge artifact(s)
2. Show the user what will be generated (file list + summary)
3. Get confirmation
4. Write files with traceability headers
5. Report what was created

### Step 4 — Sync Check

After scaffolding, verify output is current:

```bash
python3 scaffold/build_chain/cli.py sync-check
```

This parses source headers in scaffold files and compares versions against
the current knowledge artifacts. Flags stale output that needs regeneration.

## Scaffold CLI Commands

| Command | Description |
|---------|-------------|
| `artifact-summary` | Scan knowledge artifacts and display a structured summary |
| `scaffold-status` | Show what has been scaffolded vs what is pending |
| `sync-check` | Detect stale scaffold output that needs regeneration |

All commands are read-safe. `artifact-summary` reads from `knowledge/` (allowed).
`scaffold-status` and `sync-check` read from both `knowledge/` and `scaffold/`.

## Traceability Header Format

Every generated file must begin with:

```
// Generated by Narwhal Scaffold Agent
// Source: [DOC_ID] v[X.Y]
// Generated: [YYYY-MM-DD]
// DO NOT EDIT — regenerate from /scaffold-* skills
```

For multi-source files (e.g., a module informed by both HLD and API):

```
// Generated by Narwhal Scaffold Agent
// Source: HLD-0001 v0.1, API-0001 v0.2
// Generated: 2026-03-01
// DO NOT EDIT — regenerate from /scaffold-* skills
```

Adapt comment syntax to the target language (`#` for Python/YAML, `//` for
JS/TS/Go, `--` for SQL, `<!-- -->` for HTML/XML).

## Output Conventions

### `scaffold/src/`

Project source code. Structure derived from HLD module layout:

```
scaffold/src/
├── [module_a]/          # From HLD system_architecture.modules
│   ├── __init__.py      # or index.ts, main.go, etc.
│   ├── routes/          # From API artifact
│   ├── events/          # From AEC artifact
│   └── models/          # From DC artifact
├── [module_b]/
└── shared/              # Cross-module utilities from TSI
```

### `scaffold/ci/`

CI/CD pipeline files derived from CICD artifact:

```
scaffold/ci/
├── ci.yml               # Main CI pipeline (GitHub Actions, GitLab CI, etc.)
├── cd.yml               # Deployment pipeline
└── quality-gates.yml    # Quality gate definitions
```

### `scaffold/config/`

Deployment and infrastructure configs from DG and TSI:

```
scaffold/config/
├── Dockerfile           # From DG container specs
├── docker-compose.yml   # From DG + TSI service topology
├── k8s/                 # From DG Kubernetes manifests
└── env.template         # Environment variable template
```

### `scaffold/db/`

Database schemas and migrations from DBAD and DC:

```
scaffold/db/
├── migrations/          # Ordered migration files
│   └── 001_initial.sql
├── models/              # ORM model stubs
└── seeds/               # Seed data templates
```

### `scaffold/tests/`

Test stubs from UT and NFTS:

```
scaffold/tests/
├── unit/                # From UT test case definitions
│   └── test_[module].py
└── performance/         # From NFTS load test specs
    └── load_[scenario].py
```

## Prerequisite Map (for scaffold ordering)

```
scaffold-project requires: HLD (and optionally TSI)
scaffold-cicd requires: CICD (and optionally DG)
scaffold-db requires: DBAD (and optionally DC)
scaffold-api requires: API (and optionally AEC, PSD)
scaffold-tests requires: UT and/or NFTS
build-status requires: no prerequisites (read-only scan)
sync-check requires: no prerequisites (read-only scan)
read-artifacts requires: no prerequisites (read-only scan)
```

If a required knowledge artifact is missing, the scaffold skill should:
1. Report what is missing
2. Recommend creating it via the knowledge branch (`/switch-knowledge` then `/sdlc-doc-intake`)
3. Not generate partial output

---

## Rules

1. **Never write to `knowledge/`.** Read-only access only.
2. **Always confirm before writing.** Present the file list, get approval.
3. **Always include traceability headers.** Every generated file must cite its source.
4. **Follow layer order.** Architecture before contracts, contracts before tests.
5. **Respect knowledge constraints.** If a doc says "PostgreSQL", don't scaffold for MySQL.
6. **Idempotent output.** Same input should produce same output. Overwrite only with confirmation.
7. **One scaffold at a time.** Don't generate the entire project without review.
